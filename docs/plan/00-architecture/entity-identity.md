# Entity Identity — Surrogate PK + Public `EntityId`

> Canonical pattern for **every** entity stored in `SlotSmartDbContext`. Read this before any task that adds a table.
>
> This pattern supersedes the older "All entity IDs are UUIDv7" rule. UUIDv7 is still the **public** identifier; we now separate it from the database's primary key.

---

## 1. Why we do this (TL;DR)

Using a 16-byte GUID as the **primary key / clustered index key** has real costs even when the GUID is sequential (UUIDv7):

| Problem | Random GUID | UUIDv7 (sequential) | Our choice: `bigint` surrogate |
|---|---|---|---|
| Page splits / fragmentation on insert | **Severe** | Mostly fixed | Fixed |
| Clustered key included in every non-clustered index (SQL Server) | 16 bytes everywhere | 16 bytes everywhere | **8 bytes everywhere** |
| FK index size | 16 bytes per row | 16 bytes per row | **8 bytes per row** |
| Join cost (16-byte vs 8-byte compares, cache locality) | Worst | Same as random | **Best** |
| Public id leaks DB row order / count | Yes | Partially (UUIDv7 has a timestamp) | No — public id is still UUIDv7 |
| External stable identifier | Yes | Yes | **Yes — `EntityId : Guid` is still UUIDv7** |

UUIDv7 fixes fragmentation but **not** the size or join-cost problems. The dual-key pattern keeps the UX of stable, opaque, externally-shareable identifiers while letting the storage engine work with small sequential keys.

## 2. The pattern in one block

```csharp
// SlotSmart.Domain/Common/Entity.cs
public abstract class Entity
{
    // Public, stable, externally shareable identifier. UUIDv7. Set in the constructor.
    public Guid EntityId { get; protected init; }

    // Domain equality is by EntityId. The surrogate PK is invisible at this layer.
    public override bool Equals(object? obj) =>
        obj is Entity other && GetType() == other.GetType() && EntityId == other.EntityId;

    public override int GetHashCode() => HashCode.Combine(GetType(), EntityId);
}
```

```csharp
// SlotSmart.Infrastructure/Persistence/Configurations/EntityConfiguration.cs
public abstract class EntityConfiguration<TEntity> : IEntityTypeConfiguration<TEntity>
    where TEntity : Entity
{
    public virtual void Configure(EntityTypeBuilder<TEntity> builder)
    {
        // Hidden surrogate primary key — added as an EF Core SHADOW property.
        // bigint, IDENTITY (SQL Server) / GENERATED BY DEFAULT AS IDENTITY (Postgres).
        builder.Property<long>("Id").ValueGeneratedOnAdd().UseIdentityColumn();
        builder.HasKey("Id");

        // Public domain identifier — UUIDv7, server-generated, never updated.
        builder.Property(e => e.EntityId)
               .ValueGeneratedNever()
               .IsRequired();

        // Unique index on the public id (also used as the join target for FKs that
        // want to reference by public id — see §5).
        builder.HasIndex(e => e.EntityId).IsUnique();
    }
}
```

That's it. Every concrete entity inherits from `Entity`; every EF configuration inherits from `EntityConfiguration<T>` and adds entity-specific mapping in its own `Configure` method (after calling `base.Configure(builder)`).

Multi-tenant entities also implement `IMultiTenantEntity` (see §4).

## 3. Naming

| Layer | Name | Type | Notes |
|---|---|---|---|
| Domain (C#) | `EntityId` | `Guid` (UUIDv7) | The only id the domain knows about. |
| Database column | `entity_id` (Postgres) / `EntityId` (SQL Server) | `uuid` / `uniqueidentifier` | Unique index, NOT NULL. |
| Surrogate PK (EF shadow) | `"Id"` | `bigint` | IDENTITY / GENERATED. Never appears in C# code. |
| Database column | `id` / `Id` | `bigint` | Clustered PK (SQL Server) or PK B-tree (Postgres). |
| Public JSON | `id` | string (UUID) | We serialize `EntityId` as `id` on the wire so REST clients see what they expect. |
| URL paths | `/api/v1/members/{id}` | UUID | Always the `EntityId`. |
| JWT `sub`, audit `actor.userId`, outbox event payloads | UUID | `EntityId` only. The surrogate **never** leaves the DB. |

JSON property mapping is configured globally via `JsonSerializerOptions` or `[JsonPropertyName("id")]` on DTOs:

```csharp
public sealed record MemberDto(
    [property: JsonPropertyName("id")] Guid EntityId,
    string FirstName,
    string LastName,
    ...
);
```

## 4. Multi-tenant entities

```csharp
public interface IMultiTenantEntity
{
    // Surrogate FK to Tenants.Id (long). Used by the EF Core global query filter.
    long TenantId { get; }
}
```

- `TenantId` on multi-tenant tables is `bigint` (FK to `Tenants.Id`, the surrogate).
- The Guid form of the tenant (`Tenant.EntityId`) is **not** stored on per-row tables — it would just be wasted bytes on every row.
- The EF Core global query filter compares against `ITenantContext.TenantId` (a `long`), not the Guid.
- The `TenantStampingInterceptor` reads `ITenantContext.TenantId` and stamps it on insert.

This means the `ITenantContext` resolved in `phase-2-auth-tenancy/task-01-tenant-resolution.md` carries **both**:

```csharp
public interface ITenantContext
{
    long TenantId { get; }            // surrogate; used everywhere internally
    Guid TenantEntityId { get; }      // UUIDv7; used in JWT, audit, public URLs
    string TenantSlug { get; }
    bool IsResolved { get; }
}
```

JWT validation resolves `tenant_id` (a Guid claim) → `Tenants.Id` (a long) via an in-memory cache (tenants are small in number and change rarely). Cache invalidation: on tenant suspension/deletion, evict the cache entry.

## 5. Foreign keys

Two options exist; we pick option A as the default.

### A. Surrogate FKs (default — full perf benefit)

FK columns reference the surrogate `bigint` of the parent table. Domain code uses **navigation properties** to express relationships.

```csharp
// Domain
public sealed class Booking : Entity, IMultiTenantEntity
{
    public long TenantId { get; private init; }

    // Navigation properties — set by handlers; EF derives FK columns on save.
    public Training Training { get; private set; } = default!;
    public Member Player { get; private set; } = default!;
    public Member BookedBy { get; private set; } = default!;

    // Public ids exposed for events / DTOs (computed via navigation).
    public Guid TrainingEntityId => Training.EntityId;
    public Guid PlayerEntityId => Player.EntityId;
    // ...
}
```

```csharp
// EF Configuration
public sealed class BookingConfiguration : EntityConfiguration<Booking>
{
    public override void Configure(EntityTypeBuilder<Booking> builder)
    {
        base.Configure(builder);

        builder.ToTable("Bookings");

        builder.HasOne(b => b.Training)
               .WithMany()
               .HasForeignKey("TrainingId")   // shadow bigint FK
               .OnDelete(DeleteBehavior.Restrict);

        builder.HasOne(b => b.Player)
               .WithMany()
               .HasForeignKey("PlayerId")
               .OnDelete(DeleteBehavior.Restrict);

        builder.HasOne(b => b.BookedBy)
               .WithMany()
               .HasForeignKey("BookedById")
               .OnDelete(DeleteBehavior.Restrict);

        // Multi-tenant FK to Tenants.Id (bigint).
        builder.Property(b => b.TenantId).IsRequired();
        builder.HasIndex(b => new { b.TenantId, b.Training });   // example composite
    }
}
```

Handlers do the **one-time Guid → entity lookup** at the boundary:

```csharp
public async Task<Result<BookingDto>> Handle(CreateBookingCommand cmd, CancellationToken ct)
{
    var training = await _db.Trainings
        .SingleOrDefaultAsync(t => t.EntityId == cmd.TrainingEntityId, ct);
    if (training is null) return Errors.TrainingNotFound;

    var player = await _db.Members
        .SingleOrDefaultAsync(m => m.EntityId == cmd.PlayerEntityId, ct);
    if (player is null) return Errors.PlayerNotFound;

    var booking = Booking.Create(training, player, bookedBy: actor, _clock.UtcNow);
    _db.Bookings.Add(booking);
    await _db.SaveChangesAsync(ct);

    return new BookingDto(booking.EntityId, ...);
}
```

The unique index on `EntityId` makes these lookups O(log n) and extremely cheap; both lookups are on hot indexes that fit in cache.

### B. Public-id FKs (escape hatch, NOT default)

If a table will see many bulk inserts that already have the Guid in hand and no parent entity load is wanted, FKs can reference `EntityId` instead, via `HasPrincipalKey(e => e.EntityId)`. This is allowed only with a documented justification per table — examples that **might** qualify: `OutboxMessages`, `AuditEntries`. Even then, the table still has its own surrogate `Id` as PK; only the **FK** column type changes.

Default everywhere else: option A.

## 6. Identity tables (ASP.NET Core Identity)

ASP.NET Core Identity is generic on the key type. We use `long`:

```csharp
public sealed class ApplicationUser : IdentityUser<long>
{
    public Guid EntityId { get; init; } = UuidV7.NewGuid();
    public long TenantId { get; init; }   // multi-tenant
    public bool IsActive { get; set; }
    // ... domain-relevant fields
}
```

- Identity PK is `long`.
- `EntityId : Guid` is a separate non-clustered unique column.
- JWT `sub` = `EntityId`.

`UserManager<ApplicationUser>` works unchanged; lookups by Guid go through `userManager.Users.SingleOrDefaultAsync(u => u.EntityId == guid, ct)`.

## 7. Outbox, audit, and other "hot tables"

- `OutboxMessages.Id : bigint` PK, surrogate.
- `OutboxMessages.EntityId : Guid` — optional; on hot append-only tables we may omit `EntityId` entirely because the row is never referenced externally. Document per table.
- `AuditEntries.Id : bigint` PK; `EntityId : Guid` present (audit entries are queried by id from admin UI).
- `Notifications.Id : bigint` PK; `EntityId : Guid` present (clicked from the UI).

Rule of thumb: **expose `EntityId` only when the row is referenced from outside the DB.** Truly internal rows (e.g. outbox messages) can skip it and save 16 bytes per row.

## 8. Database specifics

### PostgreSQL (primary)

- Tables are heap-organized. The PK is a unique B-tree on `id` (bigint).
- `id bigint GENERATED BY DEFAULT AS IDENTITY` is preferred over `bigserial` (modern SQL standard).
- `entity_id uuid NOT NULL` with a `UNIQUE INDEX` (B-tree).
- Sort the table physically by `id` once via `CLUSTER` only if profiling shows benefit; usually not needed.
- Indexes do **not** carry the PK (Postgres uses CTID), so the per-index savings noted in the article apply differently — but per-row FK savings still apply, and table PK savings are real.

### SQL Server (compatibility target)

- `Id bigint IDENTITY PRIMARY KEY CLUSTERED` — the table **is** the clustered index.
- Every non-clustered index implicitly carries the clustered key. With `bigint` that's 8 bytes per index row instead of 16. Significant.
- `EntityId uniqueidentifier NOT NULL` with a non-clustered `UNIQUE INDEX`.

EF Core's `UseIdentityColumn()` resolves to the right provider syntax in both engines.

## 9. Architecture tests

Add to `SlotSmart.Architecture.Tests`:

1. **Every concrete entity inherits from `Entity`.**
2. **No `[Key]` attributes on `EntityId`** — the PK is the surrogate, not the Guid.
3. **Every entity configuration calls `base.Configure(builder)`** so the surrogate is wired.
4. **No public API DTO exposes the surrogate `Id`** — search the OpenAPI document for `int64` properties named `id`.
5. **`EntityId` is set in the constructor / factory** and the property has no public setter.

## 10. Migration / refactor checklist when applying this to a task

When you pick up a task that defines an aggregate, translate as follows:

- **"`Id` (UUIDv7)"** in a task field list → becomes **`EntityId : Guid`** in the C# entity. The hidden surrogate `Id : long` is added automatically by the base class / configuration.
- **"references `MemberId` (Guid)"** → becomes a **navigation property** `Member` on the entity plus a shadow `bigint` FK in the configuration.
- **"unique on `(TenantId, EntityId)`"** is redundant — `EntityId` is globally unique on its own (UUIDv7). Use `(TenantId, ...)` composite indexes only for hot tenant-scoped queries.
- **DTO field `id`** maps to `EntityId` via `[JsonPropertyName("id")]`.

## 11. Edge cases & gotchas

- **`HasData` / seed data**: you must specify the shadow surrogate value yourself when seeding (`builder.HasData(new { Id = 1L, EntityId = ..., ... })` via anonymous projection). Alternative: seed via a migration `Up` method with raw SQL using `pg_get_serial_sequence(...)`/`NEXT VALUE FOR` — pick one, document.
- **Generic repositories** that key on `Guid Id` need rewrites — they should key on `Expression<Func<TEntity, Guid>> entityId` instead. We don't have any in the plan, but watch out.
- **Bulk inserts** (e.g. training materialization) should use `AddRange` so EF assigns sequential surrogate values via a single `INSERT ... RETURNING id` (Postgres) or `OUTPUT inserted.Id` (SQL Server) batch.
- **Bookings race & advisory lock** (P5-T01): the per-training advisory lock key uses `training.Id` (the bigint surrogate) — a clean `pg_advisory_xact_lock(:training_id)` — instead of hashing a Guid string. One less indirection.
- **Equality**: `EntityId` is the source of truth for domain equality. Do **not** compare surrogate `Id` in domain code; the surrogate is meaningless to the domain.
- **Detached entities** (created in memory but not yet saved) have `Id = 0` (default `long`) until `SaveChangesAsync`. Domain code must not rely on the surrogate being set.

## 12. What this changes in the task prompts

The 20+ task prompts in `phase-1` through `phase-6` were written before this decision. They list aggregate fields as `Id, TenantId, FieldA, FieldB, ...`. Read those as follows:

- `Id` = the public `EntityId : Guid (UUIDv7)`.
- `TenantId` = `long` (surrogate FK to `Tenants.Id`).
- Any field listed as `FooId : Guid` referring to another aggregate is a navigation + shadow `bigint` FK in the DB; the Guid form is available via `entity.Foo.EntityId`.

When in doubt, this document wins.
