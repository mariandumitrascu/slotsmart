### ADR-007: Dual-Key Identity Pattern (surrogate `bigint` PK + public `EntityId : Guid`)

**Date**: 2026-05-13
**Status**: Accepted — supersedes ADR-003
**Deciders**: Project owner (designed in `docs/plan/00-architecture/entity-identity.md`); SUPERVISOR ratified into the decision log.

#### Context
ADR-003 ("Use UUIDv7 Identifiers") committed to UUIDv7 as the entity primary key. While UUIDv7 fixes the random-GUID fragmentation problem, it does **not** address two real costs:
- 16 bytes per FK / per non-clustered-index row (vs 8 bytes for `bigint`). On SQL Server every NCI implicitly carries the clustered key — so every NCI pays the full 16-byte cost on every row.
- Join cost: 16-byte vs 8-byte comparisons + cache-line locality.

For a multi-tenant SaaS with hot tables (bookings, attendance, outbox, audit) these costs compound.

#### Decision
Adopt a **dual-key pattern** for every entity stored in `SlotSmartDbContext`:
- A hidden `bigint` surrogate primary key (`Id`, EF Core shadow property; `IDENTITY` / `GENERATED BY DEFAULT AS IDENTITY`). This is the actual PK / clustered key.
- A public `EntityId : Guid` (UUIDv7) — server-generated, never updated, with a unique index. The only id the domain layer knows about; the only id that ever leaves the database (URLs, JWT claims, audit, outbox payloads, OpenAPI).
- FKs reference the surrogate `bigint` by default (option A in the pattern doc). Domain code expresses relationships via navigation properties; handlers do a one-time `Guid → entity` lookup at the boundary.
- `IMultiTenantEntity { long TenantId }` is the surrogate FK to `Tenants.Id`. The Guid form lives in `ITenantContext.TenantEntityId` and on `Tenants.EntityId`.
- An `Entity` base class lives in `SlotSmart.Domain/Common/Entity.cs`. Every concrete entity inherits from it. Architecture tests enforce this.

Canonical reference: [`docs/plan/00-architecture/entity-identity.md`](../../plan/00-architecture/entity-identity.md). When in doubt, that document wins.

#### Alternatives Considered
1. **UUIDv7 as primary key** (ADR-003 as written) — superseded: pays 16 bytes per FK and per NCI row indefinitely; SQL Server clustered-key inclusion makes this especially costly.
2. **Random GUID PK** — not considered seriously; catastrophic fragmentation.
3. **Public `bigint` ids (no Guid layer)** — rejected: leaks row order / row count; fragile for sharding / merge-import scenarios; bad UX for external API consumers.
4. **`EntityId : Guid` only on aggregates, not on internal value objects / hot append-only tables** — partially adopted: pattern §7 already allows omitting `EntityId` on truly internal rows (e.g. `OutboxMessages`).

#### Consequences
**Positive**:
- Smaller per-row indexes; smaller FKs; faster joins on hot tables.
- Public ids stay stable, opaque, externally shareable (UUIDv7).
- Clean separation: domain identity (`EntityId`) vs storage identity (`Id`).
- Multi-tenant filter compares `long` to `long` — fastest possible.

**Negative**:
- One extra column per entity (8 bytes for `Id` + the `EntityId` is already there).
- One extra unique index on `EntityId` per table.
- Handlers do a small `Guid → entity` lookup at the boundary (cached on the unique index, but still a hop).
- Seed data must specify the surrogate (`HasData(new { Id = 1L, EntityId = ... })`).
- Generic repositories that key on `Guid` need rewrites — none in the plan.

**Neutral**:
- `ApplicationUser : IdentityUser<long>` instead of `<Guid>` (Identity is generic on key type, so this is a one-line change).

#### Related Decisions
- **ADR-003** is now **Superseded by ADR-007**. UUIDv7 is still in use — but only as the public `EntityId`, not as the storage PK.
- Affects every aggregate task in P1–P5. Per the plan README's interpretation note: when a task spec says "`Id` (UUIDv7)" that means `EntityId : Guid`; "`TenantId`" means `long`; "`FooId : Guid`" referring to another aggregate is a navigation + shadow `bigint` FK.

#### Adoption checkpoints
- **P1-T01**: introduces `Entity` base class in Domain, `IMultiTenantEntity` with `long TenantId`, architecture tests asserting entity inheritance.
- **P1-T02**: introduces `EntityConfiguration<T>` base, surrogate-PK shadow property, `EntityId` unique index.
- **P2-T01**: `ITenantContext` carries both `long TenantId` and `Guid TenantEntityId`; query filter uses the `long`.
- **P2-T02**: `ApplicationUser : IdentityUser<long>` with separate `EntityId` column; JWT `sub` = `EntityId`.

### ADR-006: Pin .NET 10 SDK (10.0.300, GA / LTS)

**Date**: 2026-05-13
**Status**: Accepted
**Deciders**: SUPERVISOR (executing P1-T01 in hybrid mode)

#### Context
Risk R1 originally treated .NET 10 as "not GA". On 2026-05-13 the SUPERVISOR confirmed via the official release-metadata feed that **.NET 10 is GA as of 2026-05-12** (`channel-version: 10.0`, `latest-sdk: 10.0.300`, `release-type: lts`, EOL 2028-11-14). The "preview / fallback" complication is no longer needed.

#### Decision
- Install .NET 10 SDK 10.0.300 (matched runtime 10.0.8, AspNetCore 10.0.8). On the SUPERVISOR's machine the SDK was installed to `~/.dotnet` via the official `dotnet-install.sh` (no sudo needed; coexists with the existing `/usr/local/share/dotnet/` install).
- `backend/global.json` pins `version: "10.0.300"` with `rollForward: "latestFeature"` and `allowPrerelease: false`.
- Developer onboarding step (added to `README.md`): `curl -sSL https://dot.net/v1/dotnet-install.sh | bash -s -- --channel 10.0 --install-dir "$HOME/.dotnet"` followed by `export DOTNET_ROOT="$HOME/.dotnet"; export PATH="$HOME/.dotnet:$PATH"`. macOS users on Homebrew may instead run `brew install dotnet` (formula version 10.0.107 at this writing); the global.json pin will rollForward to whichever 10.0.x is present.

#### Alternatives Considered
1. **Use existing .NET 9.0.200**: rejected — plan & tech-stack call for .NET 10 LTS.
2. **brew install dotnet**: viable, but creates a third dotnet location (`/opt/homebrew/opt/dotnet/`). User-local `~/.dotnet` is cleaner and doesn't require sudo.
3. **System-wide install via Microsoft .pkg installer**: viable for production developer machines; documented in README as an alternative.

#### Consequences
**Positive**: matches plan; reproducible; LTS support to 2028-11-14; no preview-bug risk.
**Negative**: developers must install .NET 10 SDK alongside any existing install. Documented mitigations: brew, install script, .pkg installer.

#### Related Decisions
- ADR-002 (.NET 10 + Clean Architecture) — confirmed.
- Closes risks R1 and R2 in `EXECUTION-PLAN.md`.

### ADR-005: Worker Handoff Prompts Generated Just-in-Time

**Date**: 2026-05-12
**Status**: Accepted
**Deciders**: SUPERVISOR (subject to user override)

#### Context
The mission is to execute Phases 1–5 (33 tasks). Each handoff prompt embeds context like "what's been done so far", "git sha at handoff", and "your dependencies are met because…". Pre-generating all 33 prompts up front would (a) bloat the supervisor session immediately, (b) guarantee that prompts go stale before workers read them.

#### Decision
- Maintain a permanent `handoff-prompts/_template.md` and `handoff-prompts/README.md`.
- Pre-generate **only** the prompt for the immediately-unblocked task.
- Generate the next batch of prompts at the moment a dependency resolves (e.g., when P1-T01 is accepted, generate P1-T02/T04/T05/T07 in the same session).

#### Alternatives Considered
1. **Pre-generate all 33** — rejected: stale-prompt risk, supervisor context bloat.
2. **No prompts; workers just read the task file directly** — rejected: loses the framework context, "what's been done", and supervision contract.

#### Consequences
**Positive**: prompts are always current; supervisor context stays small; pattern matches the framework's worker-context-isolation intent.
**Negative**: each phase advancement has a small SUPERVISOR overhead to generate the next prompts.

### ADR-004: Phase Gates as Checked-in Markdown with Executable Commands

**Date**: 2026-05-12
**Status**: Accepted
**Deciders**: SUPERVISOR (subject to user override)

#### Context
The user requested verification & testing between phases. The framework needs a clear, repeatable contract for when a phase is "done" and the next phase can open.

#### Decision
- Create `docs/SUPERVISOR/phase-gates/phase-{1..5}-gate.md`, one per phase boundary.
- Each gate doc contains: pre-conditions (task list), automated verification (shell commands with expected outputs), manual verification (observable behaviors), promotion checklist, and an append-only run log.
- A phase is **closed** only when its gate passes per `EXECUTION-PLAN.md` §2.2.

#### Alternatives Considered
1. **CI-only gates (e.g., `tools/phase-gate.sh`)** — rejected for now: the gates evolve with the project; markdown is easier to amend in PRs and can be promoted to scripts later.
2. **Implicit gates (just check the phase README acceptance criteria)** — rejected: not actionable enough; no run log.

#### Consequences
**Positive**: anyone can verify a phase from a clean clone by running the listed commands; PR reviewers can spot gate-coverage gaps.
**Negative**: the gate spec must be kept in sync with the plan when task scope changes (mitigated by version-stamping each gate doc).

### ADR-001: Adopt AI Supervisor-Worker Framework

**Date**: 2026-05-12 
**Status**: Accepted 
**Deciders**: Project Team

#### Context
Need a structured approach to AI-assisted development that enables:
- Clear task delegation
- Quality assurance
- Knowledge preservation
- Session continuity

SlotSmart is a complex SaaS platform with multiple components (backend API, frontend, database, authentication, multi-tenancy) that benefits from organized AI coordination.

#### Decision
Adopt the Supervisor-Worker Framework for AI collaboration, with framework files stored in `docs/SUPERVISOR/`.

#### Alternatives Considered
1. **Ad-hoc AI usage** - No structure, just ask questions as needed
   - Rejected: Leads to inconsistent results, poor handovers on a complex project
2. **Custom framework** - Build our own from scratch
   - Rejected: Time investment, reinventing the wheel
3. **No AI assistance** - Traditional development only
   - Rejected: Miss productivity benefits

#### Consequences
**Positive**:
- Consistent AI collaboration patterns
- Clear accountability for tasks
- Preserved decision history
- Methodology skills (TDD, debugging, etc.) available as project-level Cursor skills

**Negative**:
- Some overhead in maintaining documents
- Learning curve for the framework

### ADR-003: Use UUIDv7 Identifiers

**Date**: 2026-05-12
**Status**: **Superseded by ADR-007 (2026-05-13)** — UUIDv7 is retained as the public `EntityId` only; the storage primary key is a `bigint` surrogate.
**Deciders**: Project Team

#### Context
Entity identifier strategy for all domain entities.

#### Decision (original — partially superseded)
Use UUIDv7 (time-ordered UUID) for all entity identifiers.

#### Consequences
**Positive**:
- Time-ordered for database index efficiency
- Globally unique without central coordination
- Supports multi-tenant scenarios well

**Negative**:
- Larger than integer IDs (16 bytes vs 4 bytes)
- Requires UUID generation library

#### Why this was superseded
ADR-007 keeps UUIDv7 as the **public** `EntityId` (URLs, JWT, audit, OpenAPI) but moves the storage / clustered primary key to a hidden `bigint` surrogate. This avoids the per-FK / per-NCI 16-byte cost UUIDv7 cannot fix. See ADR-007 for the full pattern.
